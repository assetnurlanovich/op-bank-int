# Outbound Kaspi Contracts

## 1. Назначение

Этот документ фиксирует недостающие исходящие контракты OnePay -> Kaspi, без которых агент не может детерминированно реализовать:
- регистрацию invoice / payment на стороне Kaspi;
- fallback-проверку статуса платежа;
- формирование исходящей подписи;
- обработку ответов и ошибок банка.

Если официальный endpoint path Kaspi отличается между средами или версиями интеграции, код не должен хардкодить путь.
Фактические `base_url`, `path`, `timeout` и, при необходимости, дополнительные headers должны приходить из конфигурации.

---

## 2. Общие правила для исходящих вызовов

- transport: HTTPS + JSON;
- все вызовы подписываются по тем же правилам нормализации, что и входящие запросы;
- `merchant_id` обязателен;
- `request_time` обязателен и имеет формат `YYYYMMDDHHMMSS`;
- `sign` обязателен;
- для денежных полей используется decimal-арифметика;
- все raw request/response payload должны сохраняться в audit log;
- timeout каждого вызова должен быть конфигурируемым;
- retry допустим только для технических ошибок и только для идемпотентных операций.

### 2.1. Стандартный envelope ответа Kaspi

- если Kaspi использует свой собственный envelope, адаптер должен преобразовать его к внутренней модели below:

```json
{
  "success": true,
  "data": {},
  "error_code": 0,
  "message": "OK"
}
```

- `success` -> признак успешной обработки запроса банком;
- `data` -> полезная нагрузка;
- `error_code` -> код ошибки банка или `0`;
- `message` -> текстовое описание.

## 3. Операция A - Register Invoice / Payment

## 3.1. Назначение

Используется для сценария, когда платеж стартует на стороне OnePay / Ломбарда, а клиент завершает оплату в Kaspi.

## 3.2. Логический client contract

Рекомендуемый метод адаптера в коде:
`KaspiClient.RegisterInvoice(ctx, request) (response, error)`

HTTP method:
`POST`

HTTP path:
конфигурируемый, например `KASPI_INVOICE_PATH`.

## 3.3. Запрос

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| merchant_id | integer | да | Идентификатор мерчанта |
| payment_id | integer | да | Идентификатор платежа в OnePay |
| contract_number | string(1..50) | да | Номер договора |
| iin | string(12) | да | ИИН клиента |
| repayment_type | integer | да | `1` или `2` |
| payment_amount | number(18,2) | да | Сумма погашения без комиссии |
| fee_amount | number(18,2) | да | Комиссия |
| total_amount | number(18,2) | да | Итоговая сумма к оплате |
| currency | string(3) | да | Обычно `KZT` |
| description | string(1..255) | нет | Текст для отображения клиенту |
| return_url | string(1..255) | да | URL возврата клиента после завершения UX-сценария |
| callback_url | string(1..255) | нет | Явный callback URL, если Kaspi требует его в payload |
| expires_at | timestamp14 | нет | Время истечения invoice |
| request_time | timestamp14 | да | Время формирования запроса |
| sign | string | да | Подпись |

## 3.4. Правила валидации запроса

- `merchant_id` > 0;
- `payment_id` > 0;
- `contract_number` не пустой;
- `iin` содержит 12 цифр;
- `repayment_type` ∈ `{1, 2}`;
- `payment_amount` >= 0;
- `fee_amount` >= 0;
- `total_amount` >= 0;
- рекомендуется проверять `total_amount = payment_amount + fee_amount`;
- `currency` длиной 3 символа;
- `return_url` не пустой;
- `request_time` корректен;
- `sign` не пустой.

## 3.5. Подпись

Базовое поле подписи:
`payment_id`

Формула:
`sha512(payment_id + request_time + secret)`

## 3.6. Успешный ответ

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| merchant_id | integer | да | Идентификатор мерчанта |
| payment_id | integer | да | Идентификатор платежа в OnePay |
| reference_id | string(1..64) | да | Внешний идентификатор платежа / invoice на стороне Kaspi |
| status | string(1..30) | да | Начальный внешний статус, например `CREATED` |
| payment_url | string(1..255) | нет | Ссылка на сценарий оплаты, если Kaspi ее возвращает |
| expires_at | timestamp14 | нет | Срок действия invoice |
| message | string(1..255) | нет | Дополнительное описание |

## 3.7. Ошибочный ответ

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| merchant_id | integer | да | Идентификатор мерчанта |
| payment_id | integer | да | Идентификатор платежа |
| error_code | integer | да | Код ошибки банка |
| message | string(1..255) | да | Текст ошибки |
| retryable | boolean | нет | Технический признак, определяемый адаптером OnePay |

## 3.8. Правила обработки ответа

- при `success = true` сервис обязан сохранить `reference_id` до возврата результата вызывающей стороне;
- при технической ошибке или timeout допускается controlled retry;
- при бизнес-ошибке повтор не выполняется автоматически;
- invoice creation не должна переводить платеж в состояние "оплачен";
- после успешной регистрации платеж должен перейти в состояние ожидания внешней оплаты.

## 4. Операция B - Get Payment Status

## 4.1. Назначение

Используется как fallback-механизм подтверждения или сверки статуса платежа.

## 4.2. Логический client contract

Рекомендуемый метод адаптера в коде:
`KaspiClient.GetPaymentStatus(ctx, request) (response, error)`

HTTP method:
`GET`

HTTP path:
конфигурируемый, например `KASPI_STATUS_PATH`.

## 4.3. Запрос

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| merchant_id | integer | да | Идентификатор мерчанта |
| payment_id | integer | нет | Внутренний идентификатор платежа |
| reference_id | string(1..64) | нет | Внешний идентификатор платежа |
| rnn | string(1..64) | нет | Внешний номер операции |
| request_time | timestamp14 | да | Время формирования запроса |
| sign | string | да | Подпись |

## 4.4. Правила валидации запроса

- `merchant_id` > 0;
- хотя бы одно из `payment_id`, `reference_id`, `rnn` должно быть передано;
- если передан `payment_id`, он должен быть > 0;
- если передан `reference_id`, он не должен быть пустым;
- если передан `rnn`, он не должен быть пустым;
- `request_time` корректен;
- `sign` не пустой.

## 4.5. Подпись

Чтобы убрать неоднозначность текущих документов, для `status` вводится явное правило выбора базового поля:
1. `payment_id`, если он передан;
2. иначе `reference_id`, если он передан;
3. иначе `rnn`.

Формула:
`sha512(signature_base + request_time + secret)`

Адаптер обязан логировать, какое именно поле использовалось как `signature_base`.

## 4.6. Успешный ответ

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| merchant_id | integer | да | Идентификатор мерчанта |
| payment_id | integer | нет | Внутренний идентификатор платежа, если известен Kaspi |
| reference_id | string(1..64) | нет | Внешний идентификатор платежа |
| rnn | string(1..64) | нет | Номер операции |
| status | integer | да | `1`, `2`, `5`, `6` |
| payment_amount | number(18,2) | нет | Сумма погашения |
| fee_amount | number(18,2) | нет | Комиссия |
| total_amount | number(18,2) | нет | Общая сумма |
| currency | string(3) | нет | Валюта |
| error_code | integer | нет | Бизнес-код ошибки банка |
| message | string(1..255) | нет | Дополнительный текст |
| processed_at | timestamp14 | нет | Время обработки на стороне Kaspi |

## 4.7. Правила обработки статуса

- `1` -> внешний платеж подтвержден;
- `2` -> платеж еще в процессе, локальный payment остается в pending-состоянии;
- `5` -> платеж отменен;
- `6` -> платеж неуспешен.

Если `status` из fallback-вызова конфликтует с уже подтвержденным `SUCCESS` callback, источником истины остается ранее принятый callback, а кейс уходит в reconciliation.

---

## 5. Требования к реализации клиента

- все исходящие вызовы идут через единый `KaspiClient`;
- raw request и raw response сохраняются в `payment_logs`;
- на timeout, 5xx и network errors адаптер возвращает техническую ошибку с признаком retryable;
- на 4xx / business error адаптер возвращает typed business error;
- все денежные поля сериализуются без потери точности;
- в тестах обязателен mock HTTP server для обоих outbound-контрактов.
