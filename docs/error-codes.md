# Error Codes

## 1. Назначение

Этот документ централизует коды ошибок внешнего API OnePay -> Kaspi.
Он нужен, чтобы агент не придумывал коды по месту и не расходился между handler, service, тестами и документацией.

---

## 2. Общие правила

- все успешные ответы возвращают `error_code = 0`;
- при `success = false` код ошибки обязателен;
- `message` должен быть безопасным для внешнего клиента и не раскрывать внутренние детали;
- внутренние stack traces и raw upstream errors остаются только в логах / audit trail.

---

## 3. Каталог кодов

| Код | Название | Где применяется | Смысл |
|-----|----------|-----------------|-------|
| `0` | `OK` | все endpoint | Успех |
| `1001` | `INVALID_REQUEST` | все endpoint | Общая ошибка валидации обязательных полей, типов или формата |
| `1002` | `CLIENT_NOT_FOUND` | `contracts` | Клиент не найден |
| `1003` | `CONTRACT_NOT_AVAILABLE` | `repayment` | Договор недоступен для оплаты |
| `1005` | `AMOUNT_BELOW_MINIMUM` | `calculate-repayment` | Введенная сумма меньше минимально допустимой |
| `1006` | `AMOUNT_ABOVE_FULL_DUE` | `calculate-repayment` | Введенная сумма превышает полную задолженность |
| `1007` | `PAYMENT_NOT_FOUND` | `payment`, `callback`, `status reconciliation` | Платеж не найден |
| `1008` | `PAYMENT_ALREADY_PROCESSED` | `payment` | Платеж уже обработан |
| `1009` | `INVALID_SIGNATURE` | все подписанные endpoint | Подпись запроса не прошла проверку |
| `1010` | `INVALID_CALLBACK` | `callback` | Callback формально некорректен |
| `1011` | `STALE_REQUEST_TIME` | все подписанные endpoint | Запрос вышел за допустимое replay window |
| `1099` | `INTERNAL_ERROR` | все endpoint | Техническая ошибка приложения |

---

## 4. Правила выбора кода

## 4.1. Валидация

- если отсутствует обязательное поле, неверный тип или формат, возвращается `1001`;
- если запрос корректен по формату, но `request_time` просрочен, возвращается `1011`;
- если подпись неверна, возвращается `1009`.

## 4.2. Callback

- если callback не прошел структурную валидацию, возвращается `1010`;
- если callback валиден, но платеж не найден, используется `1007`;
- повторный уже обработанный callback не считается ошибкой и возвращает `success = true`.

## 4.3. Технические ошибки

- ошибки базы, внутренних сервисов, таймауты upstream и неожиданные исключения маппятся в `1099`, если для сценария нет более точного бизнес-кода;
- внешний `message` не должен содержать SQL, stack trace или секреты.

---

## 5. Шаблоны message

| Код | Рекомендуемый `message` |
|-----|--------------------------|
| `1001` | `Некорректные параметры запроса` |
| `1002` | `Клиент не найден` |
| `1003` | `Договор недоступен для оплаты` |
| `1005` | `Введенная сумма меньше минимально допустимой` |
| `1006` | `Сумма превышает полную задолженность` |
| `1007` | `Платеж не найден` |
| `1008` | `Платеж уже обработан` |
| `1009` | `Неверная подпись запроса` |
| `1010` | `Некорректные параметры callback` |
| `1011` | `Запрос просрочен или отклонен по replay-защите` |
| `1099` | `Внутренняя ошибка сервиса` |

---

## 6. Требования к реализации

- в коде должен быть единый package / module с константами кодов;
- в OpenAPI и тестах должны использоваться те же значения;
- нельзя дублировать числовые литералы кодов по handler-ам;
- message для одного и того же кода должен быть консистентным во всех endpoint, если нет явной причины отклониться.
