# Request Validation and Signature Rules

## 1. Назначение

Документ определяет единые правила:
- валидации входящих запросов;
- формирования и проверки подписи `sign`;
- обработки ошибок валидации.

Эти правила обязательны для всех endpoint, описанных в `docs/api-contracts.md`:
- `contracts`
- `repayment`
- `calculate-repayment`
- `payment`
- `callback`

Те же правила нормализации значений и формирования подписи применяются к исходящим вызовам, описанным в `docs/outbound-kaspi-contracts.md`, если для конкретной операции не указано более узкое правило выбора `base_field`.

## 2. Общие правила валидации

### 2.1. Проверка обязательных полей
Если обязательное поле отсутствует, запрос отклоняется.

Если поле указано как необязательное, но передано в запросе, оно все равно должно пройти полную валидацию типа и формата.

### 2.2. Проверка типов
Каждое поле должно быть приведено к ожидаемому типу без неоднозначности.
Если приведение невозможно, запрос отклоняется.

Базовые типы:
- `integer` — целое число без дробной части;
- `number(18,2)` — десятичное число с точностью не более 2 знаков после запятой;
- `string` — строка;
- `timestamp14` — дата и время в формате `YYYYMMDDHHMMSS`.

### 2.3. Проверка строк
- строковые поля проходят trim;
- пустая строка после trim считается отсутствующим значением;
- если поле обязательное, пустое значение недопустимо;
- если поле необязательное, но передано пустым после trim, запрос должен считаться некорректным.

Для текущих контрактов это правило особенно применимо к полям:
- `contract_number`;
- `reference_id`;
- `sign`;
- `rnn`;
- `message`.

### 2.4. Проверка идентификаторов и числовых значений
- `merchant_id` должен быть целым числом > 0;
- `payment_id` должен быть целым числом > 0;
- `merchant_id` должен относиться к допустимому мерчанту;
- `reference_id` должен быть строкой;
- `rnn` должен быть строкой;
- денежные поля должны передаваться как decimal-значения, а не как float-представление бизнес-логики;
- денежные поля должны поддерживать не более 2 знаков после запятой.

Денежные поля по текущим контрактам:
- `entered_amount`;
- `monthly_due_amount`;
- `interest_amount`;
- `penalty_amount`;
- `full_due_amount`;
- `min_payment_amount`;
- `max_payment_amount`;
- `repayment_amount`;
- `payment_amount`;
- `fee_amount`;
- `total_amount`.

### 2.5. Проверка специальных строковых форматов
- `iin` должен содержать ровно 12 цифр;
- `currency`, если передается, должна быть строкой длиной 3 символа;
- `contract_number` должен соответствовать ожидаемому строковому идентификатору договора и не быть пустым;
- `reference_id` должен соответствовать строковому внешнему идентификатору операции и не быть пустым.

### 2.6. Проверка дат
`request_time` должен:
- быть в формате `YYYYMMDDHHMMSS`;
- представлять корректные дату и время;
- не быть пустым.

Пример:
- корректно: `20260316094230`
- некорректно: `2026-03-16 09:42:30`

Дополнительно сервис должен проверять допустимое временное окно запроса для защиты от replay-атак.

### 2.7. Проверка перечислений
- `repayment_type` ∈ {1, 2}
- `status` ∈ {1, 2, 5, 6}

### 2.8. Проверка денежных ограничений и согласованности полей
- если по контракту поле `entered_amount` обязательно, оно должно быть > 0;
- `payment_amount` >= 0;
- `fee_amount` >= 0;
- `total_amount` >= 0;
- для callback рекомендуется проверять, что `total_amount = payment_amount + fee_amount`;
- если передан `repayment_type`, он должен быть согласован со сценарием платежа;
- если передан `error_code`, он должен быть integer;
- если передан `message`, он не должен быть пустым после trim.

## 3. Общие правила подписи

### 3.1. Общий принцип
Подпись формируется как:

`sign = sha512(base_field + request_time + secret)`

где:
- `base_field` — основной параметр конкретного сервиса;
- `request_time` — значение параметра `request_time`;
- `secret` — секретный ключ мерчанта.

### 3.2. Алгоритм
- алгоритм хеширования: SHA-512;
- результат: hex-строка в нижнем регистре.

### 3.3. Нормализация значений перед подписью
Перед формированием строки для подписи:
- значения не должны содержать пробелы по краям;
- используется строковое представление значения без дополнительных разделителей;
- значения конкатенируются строго подряд без символов-разделителей;
- для integer используется обычное десятичное строковое представление без ведущих разделителей и форматирования;
- для string используется значение после trim;
- пустые значения не могут участвовать в подписи обязательного запроса.

Пример:
`source = "202603160010" + "20260316094230" + "my-secret"`

### 3.4. Сравнение подписи
- подпись вычисляется на стороне сервиса;
- затем сравнивается с переданным `sign`;
- по умолчанию сравнение должно быть точным, в нижнем регистре;
- case-insensitive comparison допустим только если это явно отдельно согласовано.

## 4. Правила по endpoint

### 4.1. contracts

Обязательные параметры запроса:
- `merchant_id`
- `iin`
- `request_time`
- `sign`

Правила валидации:
- `merchant_id` > 0;
- `iin` должен содержать ровно 12 цифр;
- `request_time` должен быть корректной датой и временем;
- `sign` не должен быть пустым.

Базовое поле подписи:
`iin`

Формула подписи:
`sha512(iin + request_time + secret)`

### 4.2. repayment

Обязательные параметры запроса:
- `merchant_id`
- `contract_number`
- `iin`
- `repayment_type`
- `reference_id`
- `request_time`
- `sign`

Правила валидации:
- `merchant_id` > 0;
- `contract_number` не должен быть пустым;
- `iin` должен содержать ровно 12 цифр;
- `repayment_type` ∈ {1, 2};
- `reference_id` не должен быть пустым;
- `request_time` должен быть корректной датой и временем;
- `sign` не должен быть пустым.

Базовое поле подписи:
`contract_number`

Формула подписи:
`sha512(contract_number + request_time + secret)`

### 4.3. calculate-repayment

Обязательные параметры запроса:
- `merchant_id`
- `contract_number`
- `iin`
- `payment_id`
- `entered_amount`
- `reference_id`
- `request_time`
- `sign`

Правила валидации:
- `merchant_id` > 0;
- `payment_id` > 0;
- `contract_number` не должен быть пустым;
- `iin` должен содержать ровно 12 цифр;
- `entered_amount` > 0;
- `entered_amount` должен передаваться с точностью не более 2 знаков после запятой;
- `reference_id` не должен быть пустым;
- `request_time` должен быть корректной датой и временем;
- `sign` не должен быть пустым.

Базовое поле подписи:
`payment_id`

Формула подписи:
`sha512(payment_id + request_time + secret)`

### 4.4. payment

Обязательные параметры запроса:
- `merchant_id`
- `payment_id`
- `reference_id`
- `request_time`
- `sign`

Правила валидации:
- `merchant_id` > 0;
- `payment_id` > 0;
- `reference_id` не должен быть пустым;
- `request_time` должен быть корректной датой и временем;
- `sign` не должен быть пустым.

Базовое поле подписи:
`payment_id`

Формула подписи:
`sha512(payment_id + request_time + secret)`

### 4.5. callback

Обязательные параметры запроса:
- `merchant_id`
- `payment_id`
- `status`
- `payment_amount`
- `fee_amount`
- `total_amount`
- `reference_id`
- `request_time`
- `sign`

Необязательные параметры запроса:
- `contract_number`
- `iin`
- `repayment_type`
- `error_code`
- `message`
- `rnn`

Правила валидации:
- `merchant_id` > 0;
- `payment_id` > 0;
- `status` ∈ {1, 2, 5, 6};
- `payment_amount` >= 0;
- `fee_amount` >= 0;
- `total_amount` >= 0;
- рекомендуется проверять, что `total_amount = payment_amount + fee_amount`;
- если передан `repayment_type`, то `repayment_type` ∈ {1, 2};
- если передан `contract_number`, он не должен быть пустым;
- если передан `iin`, он должен содержать ровно 12 цифр;
- если передан `rnn`, он не должен быть пустым;
- если передан `message`, он не должен быть пустым;
- `reference_id` не должен быть пустым;
- `request_time` должен быть корректной датой и временем;
- `sign` не должен быть пустым.

Базовое поле подписи:
`payment_id`

Формула подписи:
`sha512(payment_id + request_time + secret)`

## 5. Ошибки валидации

### 5.1. Отсутствует обязательный параметр
Возвращается ошибка валидации входящего запроса.

### 5.2. Некорректный формат параметра
Возвращается ошибка валидации входящего запроса.

### 5.3. Неверная подпись
Возвращается ошибка:
- `success = false`
- `error_code = 1009`
- `message = "Неверная подпись запроса"`

### 5.4. Некорректные параметры callback
Возвращается ошибка:
- `success = false`
- `error_code = 1010`
- `message = "Некорректные параметры callback"`

---

## 6. Требования к реализации

Сервис должен реализовать:
- единый validator layer;
- единый signature service;
- отдельные DTO для каждого endpoint;
- явные validation rules на уровне transport layer;
- unit tests для подписи и валидации;
- decimal-арифметику для расчетных и callback денежных полей;
- защиту от replay-атак через проверку `request_time`.

Примечание:
- исходящий контракт `status` и исходящий контракт регистрации invoice описаны в `docs/outbound-kaspi-contracts.md`;
- единые коды ошибок описаны в `docs/error-codes.md`.
