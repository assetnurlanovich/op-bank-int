# Request Validation and Signature Rules

## 1. Назначение

Документ определяет единые правила:
- валидации входящих запросов;
- формирования и проверки подписи `sign`;
- обработки ошибок валидации.

Эти правила обязательны для всех endpoint приложения OnePay ↔ Kaspi.

---

## 2. Общие правила валидации

### 2.1. Проверка обязательных полей
Если обязательное поле отсутствует, запрос отклоняется.

### 2.2. Проверка типов
Каждое поле должно быть приведено к ожидаемому типу без неоднозначности.
Если приведение невозможно, запрос отклоняется.

### 2.3. Проверка строк
- строковые поля проходят trim;
- пустая строка после trim считается отсутствующим значением;
- если поле обязательное, пустое значение недопустимо.

### 2.4. Проверка числовых значений
- `merchant_id`, `payment_id` должны быть целыми числами > 0;
- денежные поля должны быть >= 0;
- денежные поля поддерживают не более 2 знаков после запятой.

### 2.5. Проверка дат
`request_time` должен быть в формате `YYYYMMDDHHMMSS`.

Пример:
- корректно: `20260316094230`
- некорректно: `2026-03-16 09:42:30`

Дополнительно сервис должен проверять допустимое временное окно запроса для защиты от replay-атак.

### 2.6. Проверка перечислений
- `repayment_type` ∈ {1, 2}
- `status` ∈ {1, 2, 5, 6}

## 3. Общие правила подписи

### 3.1. Общий принцип
Подпись формируется как:

`sign = sha512(base_field + request_time + secret)`

где:
- `base_field` — основной параметр конкретного сервиса;
- `request_time` — значение параметра `request_time`;
- `secret` — секретный ключ мерчанта.

### 3.2. Алгоритм
- алгоритм хеширования: SHA-512
- результат: hex-строка в нижнем регистре

### 3.3. Нормализация значений перед подписью
Перед формированием строки для подписи:
- значения не должны содержать пробелы по краям;
- используется строковое представление значения без дополнительных разделителей;
- значения конкатенируются строго подряд без символов-разделителей.

Пример:
`source = "202603160010" + "20260316094230" + "my-secret"`

### 3.4. Сравнение подписи
- подпись вычисляется на стороне сервиса;
- затем сравнивается с переданным `sign`;
- рекомендуется case-insensitive comparison только если это явно согласовано;
- по умолчанию сравнение должно быть точным, в нижнем регистре.

## 4. Базовое поле подписи по сервисам

### 4.1. contracts
`base_field = iin`

Подпись:
`sha512(iin + request_time + secret)`

### 4.2. repayment
`base_field = contract_number`

Подпись:
`sha512(contract_number + request_time + secret)`

### 4.3. calculate-repayment
`base_field = payment_id`

Подпись:
`sha512(payment_id + request_time + secret)`

### 4.4. payment
`base_field = payment_id`

Подпись:
`sha512(payment_id + request_time + secret)`

### 4.5. callback
`base_field = payment_id`

Подпись:
`sha512(payment_id + request_time + secret)`

### 4.6. status
`base_field = payment_id`

Подпись:
`sha512(payment_id + request_time + secret)`

---

## 5. Ошибки валидации

### 5.1. Отсутствует обязательный параметр
Возвращается ошибка валидации входящего запроса.

### 5.2. Некорректный формат параметра
Возвращается ошибка валидации входящего запроса.

### 5.3. Неверная подпись
Возвращается ошибка:
- `success = false`
- `error_code = 1009`
- `message = "Неверная подпись запроса"`

### 5.4. Некорректные параметры callback
Возвращается ошибка:
- `success = false`
- `error_code = 1010`
- `message = "Некорректные параметры callback"`

---

## 6. Требования к реализации для агента

Агент должен реализовать:
- единый validator layer;
- единый signature service;
- отдельные DTO для каждого endpoint;
- явные validation rules на уровне transport layer;
- unit tests для подписи и валидации.
