# Data Model and State Machine

## 1. Назначение

Этот документ определяет минимальную модель хранения, по которой агент может написать:
- SQL migrations;
- repository layer;
- transaction boundaries;
- background jobs;
- idempotency и audit trail.

PostgreSQL является источником истины.
Redis используется только для ускорения технических проверок и не хранит окончательное состояние платежа.

---

## 2. Общие правила хранения

- все денежные поля в PostgreSQL хранятся как `numeric(18,2)`;
- все timestamp поля хранятся как `timestamptz`;
- все внешние raw payload сохраняются в `jsonb`;
- каждая запись должна иметь `created_at` и `updated_at`;
- для оптимистичной или пессимистичной синхронизации допускается `version` либо `SELECT ... FOR UPDATE`.

---

## 3. Таблица merchants

## 3.1. Назначение

Справочник допустимых мерчантов и их runtime-настроек.

## 3.2. Колонки

| Колонка | Тип | Ограничения |
|---------|-----|-------------|
| id | bigint | PK |
| code | varchar(50) | unique, null |
| short_name | varchar(100) | not null |
| name | varchar(255) | not null |
| email | varchar(255) | null |
| secret | varchar(255) | null |
| active | boolean | not null default true |
| replay_sec | integer | null default 300 |
| bank_url | varchar(255) | null |
| created_at | timestamptz | not null |
| updated_at | timestamptz | not null |

## 3.3. Примечание

- `short_name` нужен как короткое отображаемое имя мерчанта в UI, логах и операционных сценариях;
- `code` остается опциональным техническим кодом для внешних интеграций, routing или миграции со старых систем;
- `secret` допускается хранить в таблице для local/test и отдельных упрощенных сценариев, но для production предпочтителен secret manager или env-injection;
- `replay_sec` хранит допустимое окно replay-защиты в секундах; если значение не задано явно, используется default `300`;
- `bank_url` сделан generic, чтобы модель подходила не только для Kaspi, но и для будущих банков; если поле не заполнено, используется URL из конфигурации банковского адаптера текущей среды;
- `email` опционален и нужен для операционных уведомлений, контактных сценариев или маршрутизации алертов по мерчанту.

## 3.4. Таблица merchant_fees

## 3.4.1. Назначение

Комиссии мерчанта лучше хранить не в `merchants`, а в отдельной таблице, потому что:
- у одного мерчанта может быть несколько правил комиссии;
- комиссия может отличаться по банку;
- комиссия может отличаться по `repayment_type`;
- тарифы могут меняться во времени, и историю нужно сохранять.

## 3.4.2. Колонки

| Колонка | Тип | Ограничения |
|---------|-----|-------------|
| id | bigint | PK |
| merchant_id | bigint | not null, FK -> merchants.id |
| bank_code | varchar(30) | not null |
| repay_type | integer | null |
| fee_rate | numeric(7,4) | not null |
| fee_min | numeric(18,2) | null |
| fee_max | numeric(18,2) | null |
| currency | varchar(3) | null |
| date_from | timestamptz | not null |
| date_to | timestamptz | null |
| active | boolean | not null default true |
| created_at | timestamptz | not null |
| updated_at | timestamptz | not null |

## 3.4.3. Правила хранения комиссии

- `fee_rate` хранится в процентах, а не в долях: например, `2.5000` означает `2.5%`;
- `repay_type = null` означает общее правило для всех типов погашения;
- если есть и общее правило, и более специфичное правило по `repay_type`, приоритет у более специфичного;
- интервалы `[date_from, date_to)` не должны пересекаться для одной и той же комбинации `(merchant_id, bank_code, repay_type, currency)`;
- если процентной комиссии недостаточно, допускается одновременно использовать `fee_min` и `fee_max`.

## 3.4.4. Индексы и ограничения

- check constraint: `fee_rate >= 0`;
- check constraint: `fee_min >= 0`, если значение задано;
- check constraint: `fee_max >= 0`, если значение задано.

## 4. Таблица payments

## 4.1. Назначение

Главная агрегатная таблица платежа OnePay.
Она хранит только собственное интеграционное состояние OnePay и не дублирует бизнес-данные Ломбарда, которые могут быть получены по API Ломбарда.

## 4.2. Колонки

| Колонка | Тип | Ограничения | Описание |
|---------|-----|-------------|----------|
| id | bigint | PK | Уникальный идентификатор платежа в OnePay. Именно это значение используется как `payment_id` во внешних API OnePay ↔ Kaspi |
| merchant_id | bigint | not null, FK -> merchants.id | Идентификатор мерчанта / ломбарда |
| source_id | varchar(64) | null | Идентификатор платежа в системе Ломбарда, если такая сущность в Ломбарде существует |
| status | integer | not null default 2 | Текущий статус платежа в OnePay по кодам Kaspi |
| type | integer | not null | Тип платежа |
| bank_id | varchar(64) | null | Внешний идентификатор операции/платежа на стороне банка. Во внешнем Kaspi API этому соответствует поле `reference_id` |
| rnn | varchar(64) | null | Внешний номер операции банка, если передан банком |
| amount | numeric(18,2) | null | Полная сумма платежа, то есть `total_amount` во внешних API |
| fee | numeric(18,2) | null | Комиссия банка / мерчанта |
| currency | varchar(3) | null | Валюта платежа |
| return_url | varchar(255) | null | URL возврата клиента, если был получен от Ломбарда |
| created_at | timestamptz | not null | Время создания локальной записи |
| updated_at | timestamptz | not null | Время последнего обновления локальной записи |

## 4.3. Правила хранения

- `id` является собственным идентификатором платежа OnePay и не совпадает с идентификатором Ломбарда;
- `source_id` опционален и нужен только как ссылка на платеж Ломбарда, если Ломбард действительно возвращает собственный id;
- `bank_id` используется только как внутреннее имя банковского идентификатора; во внешних контрактах Kaspi поле по-прежнему называется `reference_id`;
- `status` хранится как integer-код по тем же значениям, что и в контрактах Kaspi;
- `type` хранится как integer-код; для текущей Kaspi-интеграции все создаваемые платежи имеют тип `4`;
- `amount` хранит полную сумму платежа, то есть `total_amount`;
- сумма без комиссии не хранится отдельно и при необходимости вычисляется как `amount - fee`;
- `contract_number`, `iin` и `repayment_type` не хранятся в `payments`, потому что это канонические данные Ломбарда;
- для получения бизнес-параметров платежа OnePay при необходимости обращается в API Ломбарда;
- результат внутреннего posting не должен вводить отдельный дополнительный код `status`, а фиксируется через `payment_logs`.

## 4.4. Индексы и ограничения

- уникальный индекс на `(merchant_id, bank_id)` where `bank_id is not null`;
- уникальный индекс на `(merchant_id, rnn)` where `rnn is not null`;
- уникальный индекс на `(merchant_id, source_id)` where `source_id is not null`;
- индекс на `(merchant_id, status)`;
- индекс на `(merchant_id, type)`;
- check constraint: `amount >= 0`, если значение задано;
- check constraint: `fee >= 0`, если значение задано.

## 4.5. Допустимые значения `status`

- `1` — `SUCCESS`, платеж успешно подтвержден банком;
- `2` — `PENDING`, платеж создан и ожидает результата от банка;
- `5` — `CANCELLED`, платеж отменен;
- `6` — `FAILED`, платеж завершился неуспешно.

## 4.6. Допустимые значения `type`

- `0` — прием;
- `1` — выплата;
- `2` — рекуррентный прием;
- `3` — рекуррентная выплата;
- `4` — Kaspi-платеж.

Для текущей системы все платежи, создаваемые в рамках данного приложения, должны иметь `type = 4`.

## 5. Таблица payment_links

## 5.1. Назначение

Минимальная таблица соответствия OnePay <-> Ломбард.
Она нужна, потому что `iin` и `contract_number` являются каноническими данными Ломбарда, но локальный платеж OnePay все равно должен быть связан с конкретным бизнес-контекстом Ломбарда.

## 5.2. Колонки

| Колонка | Тип | Ограничения | Описание |
|---------|-----|-------------|----------|
| id | bigint | PK | Технический идентификатор записи связи |
| payment_id | bigint | not null, FK -> payments.id | Идентификатор платежа OnePay |
| merchant_id | bigint | not null | Идентификатор мерчанта / ломбарда |
| iin | varchar(12) | not null | ИИН клиента в Ломбарде |
| contract_number | varchar(50) | not null | Номер договора в Ломбарде |
| source_id | varchar(64) | null | Идентификатор платежа в Ломбарде, если появился |
| created_at | timestamptz | not null | Время создания связи |
| updated_at | timestamptz | not null | Время обновления связи |

## 5.3. Правила связи

- для MVP соответствие OnePay и Ломбарда определяется по `(merchant_id, iin, contract_number)`;
- одна комбинация `(merchant_id, iin, contract_number)` должна ссылаться только на один активный локальный платеж OnePay;
- если Ломбард возвращает собственный id платежа, он сохраняется в `source_id`;
- `source_id` не обязателен и не должен блокировать сценарии, где в Ломбарде отдельного id платежа нет.

## 5.4. Индексы и ограничения

- уникальный индекс на `(merchant_id, iin, contract_number)`;
- уникальный индекс на `(merchant_id, source_id)` where `source_id is not null`;
- уникальный индекс на `(payment_id)`.

## 6. Таблица payment_logs

## 6.1. Назначение

Единый легкий журнал платежа OnePay.
Эта таблица заменяет отдельные специализированные таблицы:
- `payment_status_history`;
- `callback_events`;
- `status_checks`;
- `posting_retries`;
- `idempotency_keys`;
- старый `integration_logs`.

Такой вариант выбран как минимально достаточный для MVP.

## 6.2. Колонки

| Колонка | Тип | Ограничения | Описание |
|---------|-----|-------------|----------|
| id | bigint | PK | Идентификатор записи лога |
| payment_id | bigint | not null, FK -> payments.id | Ссылка на платеж OnePay |
| type | integer | not null | Кто или что инициировало запись лога |
| status | integer | null | Снимок `payments.status` после события, если применимо |
| event_key | varchar(128) | null | Ключ события для корреляции и легкой идемпотентности |
| request | jsonb | null | Входной или исходящий payload запроса |
| response | jsonb | null | Ответ или результат обработки |
| error_code | varchar(50) | null | Код ошибки, если есть |
| trace_id | varchar(64) | null | Trace / correlation id |
| created_at | timestamptz | not null | Время записи лога |

## 6.3. Значения `type`

- `0` — лог самого OnePay application layer;
- `1` — inbound запрос от банка;
- `2` — outbound вызов в банк;
- `3` — outbound вызов в Ломбард;
- `4` — callback / webhook обработка;
- `5` — background job / retry / status polling;
- `6` — manual / operator / technical action.

## 6.4. Индексы и ограничения

- индекс на `(payment_id, created_at)`;
- индекс на `(payment_id, type, created_at)`;
- уникальный индекс на `(payment_id, type, event_key)` where `event_key is not null`.

## 6.5. Что заменяет эта таблица

- история статусов читается по `status` + `created_at`;
- callback-аудит хранится как записи с `type = 4`;
- retry и polling хранятся как записи с `type = 5`;
- легкая идемпотентность достигается через `event_key`, если он формируется детерминированно.

## 6.6. Достаточность структуры

Для MVP структура достаточна, если:
- каждое существенное событие пишет запись в `payment_logs`;
- `status` заполняется после каждого перехода состояния;
- для callback, polling и retry формируется детерминированный `event_key`.

Если позже потребуется более глубокая операционная аналитика, можно отдельно добавить специализированные таблицы без ломки базовой модели.

## 7. Внутренние статусы платежа

| Статус | Смысл |
|--------|-------|
| `1` | `SUCCESS`, банк подтвердил успешный платеж |
| `2` | `PENDING`, платеж ожидает результата от банка |
| `5` | `CANCELLED`, платеж отменен |
| `6` | `FAILED`, банк подтвердил неуспешный результат |

---

## 8. Допустимые переходы состояний

- `2` -> `1`;
- `2` -> `5`;
- `2` -> `6`.

- Запрещено:
- переходить из `1` обратно в любой менее достоверный статус автоматически;
- терять `bank_id` или `rnn` после их сохранения;
- выполнять внутреннее проведение до фиксации внешнего подтверждения.

---

## 9. Транзакционные правила

- обработка callback выполняется в одной БД-транзакции:
  1. блокировка записи платежа;
  2. проверка business idempotency через `payment_logs.event_key`;
  3. вставка записи в `payment_logs`;
  4. обновление `payments`;
  5. вставка второй записи в `payment_logs` при необходимости фиксации результата posting.

- Если внутреннее проведение вынесено асинхронно, его запуск происходит после коммита транзакции.

---

## 10. Правила для Redis

- допустимо хранить replay-guard по `merchant_id + sign + request_time`;
- TTL должен быть меньше либо равен replay window;
- потеря Redis не должна приводить к потере факта платежа или статуса.

---

## 11. Требования к миграциям

- каждая таблица создается отдельной versioned migration;
- индексы и уникальные ограничения должны присутствовать в тех же миграциях, что и таблицы;
- обязательны down-стратегия или явно задокументированная policy forward-only;
- миграции должны быть применимы автоматически в test/dev и через controlled job в prod.
